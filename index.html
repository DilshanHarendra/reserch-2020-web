Overview

Brief description on ERT

This is a automated solution for upgrade vulnerable libraries on your Java Maven projects. You just need to update your Java Maven project source code to our application. Then we will find and fix vulnerable libraries on it. We will also make sure the library version update is completely successful by building the project. This application will save lots of time from the developers by updating the libraries manually.

Scan your Java Maven Project and find vulnerable libraries

We will scan through your project and find the vulnrable libraries. You can upload the project source code as ZIP file.

Find the higher versions of the vulnerable libraries

We will find what are the availble higher versions of each vulnearble library to select the library with minimal vulnerabilities

Select non-vulnerable libraries

We will analyze all the avaialble versions and select library versions with minumal vulnerabilities. We usually only select the libraries with CVSS Score less than 4.0

Update each library version and check for build issues

We will update each library version at a time and carefully select the compatible library versions which doesn't break the project

Provide complete summary report.

We will also provide complete report containg the updated library versions, previous versions, vulnerabilities contained and current status of the project compared with the previous satate.

Save the time of developers

We will save lots of time from the developers if they do this process manually by themselves

    Literature Survey

    SURVEY

    A. Snyk

    Synk for Java supports to check vulnerable libraries in Java projects and fix them. Users can add the project by using a source control platform like GitHub. Once the project has added, Synk will scan, direct and transitive dependencies and make a report and it will share it with the user. User can then manually check for each problem if they require. Then the user can fix the detected issues via pull/merge requests. Yet Synk does not provide a method to check build issues that might occur because of the updated dependencies.

    B. Greenkeeper

    This application will observe every module on your repository. Each time the repository updates it will check whether the updated repository doesn’t break your code. If the build breaks, Greenkeeper will notify which update to which dependency caused the issue. This can check whether any updated dependency may cause build issues. But Greenkeeper cannot identify vulnerable dependencies on the repositories.

    C. Tidelift

    Tidelift can identify unmanaged dependencies from GitHub source. It can also upgrade the library to an update version, deal with issues caused by missing or unreliable package maintainers. Reports will also generate referring to the security team of the company. Even though Tidelift is able to identify vulnerable dependencies it cannot make sure that updated libraries doesn’t have any security vulnerabilities and it can’t automatically integrate and find the compatible library version which doesn’t cause any build issues.

    D. FOSSA

    This application can manage open source components of your project. It supports most of the languages. It can resolve dependencies on your project, add usage and licensing policies to continuous integration flow, monitor security vulnerabilities on the project. FOSSA will detect the vulnerable libraries and inform them to the user. The user can then update the dependencies manually can check for integration [16]. But this application can’t upgrade the dependencies itself and make sure updated library doesn’t cause any issues. But our solution will be able to do that automatically with minimum user interaction and also if there are any build issues caused by a specific library it can roll back one version at a time and find the library version which doesn’t cause any build issues.

    Research Gap

    Up to Now

    Currently dependency upgrade process is done by the developers manually. There are tools available to perform static code analysis. But some of the tools should be manually operated whereas others only have partial implementations. Because of that most of the developers stuck into “OWASP Dependency Check” and find the available vulnerable components and update them manually. The followings steps show what is the standard (manual) procedure of updating vulnerable 3rd party libraries in majority of software companies.

    There are some additional tools available for updating vulnerable components. Most of the available tools are only capable of updating library version to the higher version in “pom.xml” file. Usually updating into the highest version is always not the optimal solution. Because it can cause build issues in the project. Currently there are no tools for do this whole process automatically with build checking.

    RESEARCH PROBLEM

    Innovation

    Doing the dependency upgrade process manually is very time consuming This process can be complex if the product contains so many vulnerable libraries. Since there are no tools for build, check and select compatible higher version of a library, developers only use OWASP Dependency check to find vulnerable libraries and versions. Because other available tools do not always do the desired task correctly.

    • Some developers do not like to update libraries because it can cause build errors
    • FIn order to do this update process successfully there should be at least few developers who have knowledge on security domain
    • Available dependency upgrade tools are only capable of doing the whole process partially
    • Updating the libraries after the product released can be quit painful and troublesome
    • Most importantly the time that developers are wasting to do this process manually can be saved for somthing else

    Research Objectives

    Goals

    • Save the developers time of doing the library upgrade process manually
    • User friendly application which can be use even with little to no knowledge on security domain
    • Make sure updated library versions don’t cause build issues and guarantee a compatible and successful library upgrade

    Methodology

    In order to address this problem we came across a web application which is capable of identifying the vulnerable dependencies from the source code of a Java project and it will automatically upgrade the dependencies for the developers and also it makes sure that there won’t be any build related issues because of updated libraries. Developers can visit the frontend of the application and upload their source code to our system. Once they upload the source code they should continue and the backend server starts to check the vulnerable components in the uploaded project. Then the system will find newer versions of those vulnerable libraries, upgrade them, check for any build issues and give the final static code analysis report of the project.

    The above figure shows the overview of our system. The application consists of 4 major parts.

    • Frontend web application
    • Backend server
    • OWASP Dependency Check
    • Maven repository
    • Jenkins server

    A. Frontend web application

    End-user must interact with the frontend web application. First end-user must visit the frontend home page and upload the Java source code to the frontend web application. The Java project must be maven. Then the End-user can upload the source code in following methods (Step 1),

    • Zip file
    • GitHub URL
    • War file

    B. Backend Server

    Backend server controls the core functions of the system. Once the uploaded project is forwarded to the backend server it will make the backup of the entire project and placed in the separate directory. Then the backend server will detect the “pom.xml” which contains the dependencies of a maven project. Backend server will then update the “pom.xml” file and add the OWASP Dependency Check plugin. Then it will send it to the OWASP Dependency Check (Step 3). Once the report of the vulnerable dependencies has been received from the Dependency Check (Step 4), the server will then find what the available newer versions of those vulnerable libraries from the maven repositories (Step 5). Then the Backend server will download all the higher versions available of each vulnerable library version to the internal server directory (Step 6). Server then runs the OWASP Dependency Check tool against that folder to find what the library versions are and what are not vulnerable. Then it will make a list of groupIDs, artifactIDs, versions which are not vulnerable libraries [13]. Then send that list along with the project (Step 7) and iteratively build each and every version and get the build details (Step 8). Then the server will make a report of library versions which are not cause any build issues. Then the front end will make the final “pom.xml” and forward it to the frontend server. End-user can then download the finalized project without vulnerable libraries. Figure
